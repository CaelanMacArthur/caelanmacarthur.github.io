Week One



Week one was figuring out the architecture and the functionality of Amazon Alexa thrid party skill development.Also, getting brought to speed on what was currently in the works for this study.


Endpoints


Amazon Alexa's architecture is proprietary I mapped out it by going to the Amazon Alexa documentation. It was found that custom skills backend can be hosted through HTTPS must be certified by a party that Amazon trusts and that list from Mozilla’s CA Certificate Program. The third-party hosting must verify that the requests are coming from the third-party skills.


Amazon Alexa's third-party skills have the option of hosting through AWS with Lambda. Alexa Lambda hosting uses end point-to-end point hosting on AWS.


Client-side (Alexa on-device)


To learn how Alexa functions it needed to learn how works. Alexa’s client-side devices run Alexa’s phone apps and Alexa’s devices on Alexa’s devices that communicate to AWS.


Server-side


On the server side, there are two different AWS services that Alexa runs on. They are Amazon DynamoDB and Amazon S3. However, if a third party is hosting through an HTTPS customize solution backend architecture in place


Amazon DynamoDB is a NoSQL based database on AWS that Alexa can do data queries to find the necessary information. Amazon S3 is a scalable data storage for enterprise and some personal usages in the AWS. Amazon S3 can region-specific if wanted. Alexa can integrate with S3 for data requests.


Amazons Security Protocols


Amazon Alexa uses two main security protocols Amazon Cognito and Amazon Cognito Pools.


Amazon Cognito lets users control access to the web and mobile applications. Amazon Cognito uses SAML 2.0 and OpenID Connect.


Amazon Cognito Pools is the functionality behind Cognito. Individuals with control of the service use Amazon Cognito Pools to assign security protocols through Alexa’s SDK.
