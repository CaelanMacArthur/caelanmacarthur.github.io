Changing Email and Record-Keeping

To scale for honeytoken testing, it was necessary to have features that changed email addresses after each skill-testing so that the email is identifiable as a specific skill. For example, if I am testing the PayPal skill and the email I am using is examp.le@gmail.com. I need examp.le@gmail.com to be correlated to PayPal.

The Python file changeEmail accomplishes these fully by using Selenium, an automation framework for web testing. Selenium functionally finds the CSS elements, Xpath’s, classes, and more in the HTML text rendered on the client-side in order to know what to target for testing. For example, I can use Selenium to find a text box that I need to know takes certain input, and I can also test that on a large scale throughout a whole website. For how powerful Selenium is, it makes sense to use this framework. changeEmail logged into the test user account through the login page, navigates to the security center, and enters in the new email when prompted.

In order to be able to go through the final step of changing the email, the Amazon Authentication system sends a one-time password to your email. This stopped Selenium from changing the password. I was able to find that the Gmail API is a public API that you can call and it returns your emails in HTML format. getMail functionality hooks up a google developer account associated with the Gmail I was using to test, gets the most recent email, parses through the HTML with beautifulsoup Python library, and finds the one-time password. Then passes the one-time password back into changeEmail so that selenium can enter the one-time password.

Additionally, so that findings can be verified it is necessary to keep detailed records of each step. In changeMail, and getUtterances, functionality for screenshots functionality was put to go along with the JSON file that voice.py was created. I used ctime to label the screenshots file so that the time date and time were labeled.

Unfortunately, since there was a dependency conflict between ctime and time libraries Python, I also needed to create a custom time method in another file where the time python library can live. Those methods are then passed as a function call to the rest of the files that use ctime for record-keeping and that also need time library functionality.

However, one of the current methods has one limitation. It only has the capability to change one combination at a time. However, this limitation is not that hard to get past. All you have to do is generate a Gmail account that is 54 characters that have 54 different versions of that email. Then have changeEmail iterate through those emails. Gmail supports up to 254, but Amazon limits characters in its new email to 54 characters when creating a new account.
