Chosen Methodology of Testing Honeytokens

The honeytokens we decided on using was a user Gmail account that is 54 characters separated by periods. Gmail ignores period when sending. For example, let’s say I had the email example@gmail.com. It would expect exa.mple@gmail.com as a valid version of that email address and still send the email to example@gmail.com.

        In order to track the data misuage we assigned a email to a skill ID. For example, if the exa.mple@gmail.com is assigned to skill ID 1234 and we got a email to exa.mple@gmail.com we know that they solid our data.

Automation Design

Due to the magnitude of third party skills we need an automation software that can scale for large sets of third-party skills. Although, we are beginning with a small section of third party skills download the testing will scale up automation software will need to be flexible enough to handle this load.

        

        As such I began the following design:

  1. Scrapes utterances from the Alexa skills store.
  2. Uses a voice synthesizer to talk to Alexa, and records Alexa’s responses.
  3. Automates changing of emails for honeytokens.
  4. Automates getting the one-time password for prompted by Amazon web portal.
  5. File management for recording utterances files (.mp3 & .wav), file management for recording screenshots while data scraping, changing emails, getting text messages for Oath, and one-time passwords prompted by Amazon’s security portal.
